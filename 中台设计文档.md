# HealthHyper — 健康数据业务中台 设计文档

> 本文档用于整理思路与规划，非正式提交物。重点围绕中台的功能设计与数据库设计展开。

---

## 一、项目简介

**项目名称**：基于 AI 的个人健康管家系统

**团队规模**：3 人毕业设计大赛项目

**整体目标**：融合穿戴式硬件、鸿蒙手机终端、云端服务与 AI 智能分析，构建端-边-云协同的个人健康管理系统。

### 1.1 三个子方向

| 子方向 | 负责人 | 核心技术 |
|---|---|---|
| 基于鸿蒙的多模态健康数据采集和交互终端 | 队友A | HarmonyOS、MindSpore Lite |
| **基于 Spring Boot 的健康数据业务中台** | **本人** | **Spring Boot、MySQL、Redis、RESTful API** |
| 面向健康管理的端-边-云协同 AI 服务 | 队友B | LangChain、RAG、大模型 |

### 1.2 系统拓扑

```
硬件(穿戴设备/MCU) --蓝牙--> 手机(鸿蒙APP) --HTTP--> 云端中台(Spring Boot) --HTTP--> 云端AI(LangChain)
```

中台在架构中的角色：**数据枢纽 + 业务引擎**。所有健康数据在中台落库，所有业务规则在中台执行，对上游（手机/硬件）和下游（云AI）提供统一的 RESTful 接口。

## 二、功能模块总览

从中台视角出发，系统功能可划分为以下模块：

| # | 模块 | 简述 | 中台职责 |
|---|---|---|---|
| 1 | 用户与认证 | 注册、登录、JWT 鉴权 | 账户管理、Token 签发与校验 |
| 2 | 健康档案 | 用户画像、个人基线、病史摘要、生活习惯 | 档案 CRUD、历史变更记录 |
| 3 | 实时生理数据 | 心率/血氧/体温等连续采集数据 | 接收存储、聚合统计（夜间均值等） |
| 4 | 基线管理 | 个人生理基线的存储与下发 | 存储硬件/AI 计算的基线，按需下发 |
| 5 | 睡眠记录 | 睡眠分期结果与睡眠报告 | 存储分期数据、计算夜间聚合指标、供 AI 分析 |
| 6 | 用药管理 | 用药计划、提醒、依从记录 | 计划 CRUD、提醒调度、依从率统计 |
| 7 | 医疗记录 | 体检报告、诊断、处方等 | 结构化文本存储（不存图片）、供 AI 检索 |
| 8 | 日常记录 | 饮食、运动、情绪、症状等主观记录 | 通用日记 CRUD、健康计划打卡 |
| 9 | 健康计划 | AI 生成的健康建议与行动计划 | 计划存储、打卡记录（复用日常记录） |
| 10 | AI 对话 | 用户与 AI 的健康咨询 | 会话管理、消息持久化、流式 SSE 透传 |
| 11 | 异常预警 | 基于数据的异常告警与 AI 主动关怀 | 接收硬件 flag、防抖规则、触发 AI 主动对话 |
| 12 | 设备管理 | 穿戴设备的简易管理 | 设备绑定/解绑、状态维护 |

## 三、数据流设计

### 3.1 已理清：硬件 & 睡眠相关数据流

#### 流向 A：硬件 → 手机 → 中台（实时生理数据）

手机作为硬件的网络代理，透传实时数据包到中台。

| 字段 | 说明 |
|---|---|
| `hr_now` | 当前心率 |
| `spo2_now` | 当前血氧 |
| `bt_now` | 当前体温 |
| `activity` | 活动量（睡眠相关） |
| `sdann` | 心率变异性指标 |
| `hr_cv` | 心率变异系数 |
| `flag` | 硬件端异常标记 |

> 传输频率：约每 30 秒一包。

#### 流向 B：硬件 → 手机 → 中台（基线数据）

硬件计算出个人基线后发送，非固定频率。

| 字段 | 说明 |
|---|---|
| `hr_base` | 心率基线 |
| `spo2_base` | 血氧基线 |
| `bt_base` | 体温基线 |
| `hr_cv_base` | 心率变异系数基线 |
| `sdann_base` | SDANN 基线 |

#### 流向 C：云 AI → 中台 → 手机 → 硬件（配准基线下发）

云 AI 分析后可能校准基线，中台负责存储并向下游下发。字段同流向 B。

#### 流向 D：手机 → 中台（睡眠分期数据）

手机端完成 MindSpore Lite 推理后，上传分期结果。

| 字段 | 说明 |
|---|---|
| `stage` | 睡眠阶段：`wake` / `n1` / `n2` / `n3` / `rem` |
| `start_time` | 该阶段起始时间 |
| `duration_sec` | 持续秒数 |

#### 流向 E：中台 → 云 AI（睡眠分析请求）

中台聚合整晚数据后，组装分析请求发给云 AI。

| 类别 | 字段 |
|---|---|
| 分期统计 | 各阶段总时长、占比 |
| 心率 | `night_avg_hr`, `night_max_hr`, `night_min_hr` |
| 血氧 | `night_avg_spo2`, `night_max_spo2`, `night_min_spo2` |
| 体温 | `night_avg_bt`, `night_max_bt`, `night_min_bt` |

### 3.2 已明确：AI 对话流式数据流

全链路流式透传，中台作为 SSE 中继：

```
手机 POST /api/chat/{sessionId}/send
  body: { "text": "...", "img_base64": "可选，data:image/jpeg;base64,..." }
       │
       ▼
中台：存 user message → 组装上下文 → POST 云AI /chat/stream
  body: { "text": "组装后的prompt", "img_url": "如有则透传base64" }
       │
       ▼
云AI 流式响应：逐段返回 {"content": "..."} ，结束发 [DONE]
       │
       ▼
中台：逐段通过 SSE 推给手机（SseEmitter），同时拼接完整回复
       │
       ▼
收到 [DONE] → 中台将完整回复存入 chat_message(role=assistant) → 关闭 SSE
```

> 中台用 Spring Boot 的 `SseEmitter` 实现。手机端消费 SSE 事件流实时展示打字效果。

### 3.3 已明确：医疗记录图片处理流

不存储图片文件，图片仅作为 AI 提取的临时输入：

```
用户拍照 → 手机将图片转 base64
  → 发送到中台（作为 AI 对话消息的附件）
    → 中台转发 base64 给云 AI
      → AI 识别提取结构化文本
        → 中台只存提取后的文字内容到 medical_record
```

> base64 编码不落库，只在请求链路中临时传递。图片的价值最终体现为 AI 提取的结构化文本。

### 3.4 待整理：手机端其他数据流

| 流向 | 场景 | 待确认内容 |
|---|---|---|
| 手机 → 中台 | 用户手动录入（用药、饮食、症状、情绪等） | 各类表单的具体字段 |
| 手机 → 中台 | 健康计划打卡 | 打卡交互细节 |
| 中台 → 手机 | 数据查询响应、告警推送 | 响应格式统一 |

**各 `log_type` 的字段定义**：

##### diet — 饮食记录

`content` 存自由文字描述（如"吃了一碗面"），`extra_json` 存结构化信息：

```json
{
  "meal_type": "lunch",
  "food_items": [
    {"name": "米饭", "amount": "1碗"},
    {"name": "西红柿炒蛋", "amount": "1份"}
  ]
}
```

| extra_json 字段 | 类型 | 说明 |
|---|---|---|
| `meal_type` | string | `breakfast` / `lunch` / `dinner` / `snack` |
| `food_items` | array | 食物条目，每项含 `name`（名称）和 `amount`（量，选填） |

> 不做热量计算，那太复杂了。用户能记录吃了什么就够用。

##### exercise — 运动记录

`content` 存自由描述，`extra_json`：

```json
{
  "exercise_type": "walking",
  "duration_min": 30,
  "intensity": "medium"
}
```

| extra_json 字段 | 类型 | 说明 |
|---|---|---|
| `exercise_type` | string | `walking` / `running` / `cycling` / `swimming` / `gym` / `other` |
| `duration_min` | int | 运动时长（分钟） |
| `intensity` | string | `low` / `medium` / `high` |

##### mood — 情绪记录

`content` 存用户想说的话，`extra_json`：

```json
{
  "mood_score": 3,
  "tags": ["tired", "anxious"]
}
```

| extra_json 字段 | 类型 | 说明 |
|---|---|---|
| `mood_score` | int | 情绪评分 1-5（1=很差, 5=很好） |
| `tags` | array | 情绪标签：`happy` / `calm` / `tired` / `anxious` / `stressed` / `sad` / `angry` 等 |

##### symptom — 症状记录

`content` 存详细描述，`extra_json`：

```json
{
  "symptom_name": "headache",
  "severity": 2,
  "duration_desc": "持续约2小时"
}
```

| extra_json 字段 | 类型 | 说明 |
|---|---|---|
| `symptom_name` | string | 症状名：`headache` / `stomachache` / `dizziness` / `fatigue` / `insomnia` / `other` |
| `severity` | int | 严重程度 1-3（1=轻微, 3=严重） |
| `duration_desc` | string | 持续时间描述（自由文本） |

##### plan_checkin — 健康计划打卡

`content` 存当日备注（选填），`extra_json`：

```json
{
  "plan_id": 1,
  "checked_items": [0, 1, 3],
  "unchecked_items": [2]
}
```

| extra_json 字段 | 类型 | 说明 |
|---|---|---|
| `plan_id` | int | 关联 health_plan |
| `checked_items` | array | 完成的计划项下标 |
| `unchecked_items` | array | 未完成的计划项下标 |

> 一天一条打卡记录，数字对应 `health_plan.items_json` 中各项的 index。

#### health_plan — 健康计划

> AI 生成的健康建议与行动计划，类似用药计划但更轻量。用户可以每日打卡跟踪执行情况（打卡记录存在 daily_log 中）。

| 字段 | 类型 | 说明 |
|---|---|---|
| id | BIGINT PK | |
| user_id | BIGINT | |
| title | VARCHAR(100) | 计划标题（如"改善睡眠质量计划"） |
| items_json | JSON | 计划项数组，见下方示例 |
| source | VARCHAR(20) | 来源：`ai`（AI 建议）/ `manual`（用户自建） |
| chat_session_id | BIGINT | 若由 AI 对话生成，关联会话（可空） |
| status | TINYINT | `1`=进行中 / `0`=已归档 |
| start_date | DATE | 开始日期 |
| end_date | DATE | 结束日期（可空=持续） |
| created_at | DATETIME | |

**`items_json` 示例**：

```json
[
  {"index": 0, "desc": "每天步行30分钟", "category": "exercise"},
  {"index": 1, "desc": "睡前1小时不看手机", "category": "sleep"},
  {"index": 2, "desc": "每天喝水2000ml", "category": "diet"},
  {"index": 3, "desc": "午餐后散步10分钟", "category": "exercise"}
]
```

> **与用药管理的对比**：用药管理有精确的时间点和剂量，需要 plan + log 两张表。健康计划更宽泛（"每天走 30 分钟"），打卡只需要记录"做了/没做"，所以用一张 plan 表 + daily_log 中的 plan_checkin 类型就够了，不需要再加一张专门的打卡表。

## 五、接口设计概要

中台对外提供 RESTful API，调用方有两个：**手机端** 和 **云 AI**。由于都走 HTTP，部分接口存在复用。

### 5.1 接口分组

| 分组 | 前缀 | 调用方 | 说明 |
|---|---|---|---|
| 认证 | `/api/auth` | 手机 | 注册、登录、刷新 Token |
| 用户 | `/api/user` | 手机 | 个人信息、健康档案 |
| 设备 | `/api/device` | 手机 | 绑定、解绑、状态 |
| 生理数据 | `/api/vital` | 手机(写) / 云AI(读) | 上报实时数据、查询历史 |
| 基线 | `/api/baseline` | 手机(写+读) / 云AI(写) | 上报/查询/下发基线 |
| 睡眠 | `/api/sleep` | 手机(写+读) / 云AI(读) | 上报分期、查询睡眠报告 |
| 用药 | `/api/medication` | 手机(写+读) / 云AI(读) | 计划管理、记录、统计 |
| 医疗记录 | `/api/medical` | 手机(写+读) / 云AI(读) | 记录管理 |
| 日常记录 | `/api/daily` | 手机(写+读) / 云AI(读) | 饮食/运动/情绪/计划打卡等 |
| 健康计划 | `/api/plan` | 手机(读写) / 云AI(写) | 计划管理、打卡统计 |
| AI 对话 | `/api/chat` | 手机(读写) | 会话管理、发送消息(SSE 流式响应) |
| 告警 | `/api/alert` | 手机(写+读) | 告警上报、查询、已读 |
| 通知轮询 | `/api/notify/poll` | 手机(读) | 手机定期轮询：新的 AI 主动对话、未读告警等 |

> **关于云 AI 接口**：中台主动调用云 AI（而非云 AI 调用中台），所以中台是云 AI 的客户端。中台在需要时（对话、睡眠分析、异常触发等）组装请求调用云 AI 的 `/chat/stream` 接口，云 AI 不需要知道中台的接口地址。

### 5.2 接口调用方向

```
手机端 ──请求──> 中台 ──请求──> 云AI
手机端 <──响应── 中台 <──响应── 云AI

手机端的接口（中台暴露）：
  /api/auth, /api/user, /api/device       -- 仅手机
  /api/vital, /api/baseline, /api/sleep   -- 手机上报+查询
  /api/medication, /api/medical, /api/daily, /api/plan  -- 业务数据
  /api/chat/{sessionId}/send              -- 发消息，SSE 流式返回
  /api/notify/poll                        -- 轮询通知（AI主动对话、未读告警）

中台调用云AI的接口（云AI暴露）：
  POST /chat/stream                       -- 对话/分析，流式响应
```

> **鉴权**：手机端用 JWT 用户 Token；中台调云 AI 用服务间 API Key（Header 中传固定 token）。

## 六、中台核心业务逻辑

除了 CRUD，中台需要承担以下业务处理：

### 6.1 实时数据接收与异常处理

- 手机上报 vital_sign 数据 → 中台校验 → 入库
- 若 `flag != 0`（硬件异常标记），生成一条 `health_alert` 记录
- **异常触发 AI 主动对话**：详见 6.6 节
- 用 Redis 缓存用户最近一次生理数据，供 APP 实时展示拉取

### 6.2 睡眠数据聚合

- 手机上报睡眠分期 → 中台存入 `sleep_stage`
- 一晚结束后（手机触发或定时任务），中台从 `vital_sign` 聚合夜间心率/血氧/体温的均值、极值 → 写入 `sleep_session`
- 组装分析请求 → 调用云 AI `/chat/stream` → AI 分析结果回填 `sleep_session.ai_analysis`

### 6.3 用药提醒调度

- 根据 `medication_plan` 的 `time_slots`，生成每日待执行的提醒
- 学生项目简易实现：手机端根据计划自行设置本地闹钟/提醒，中台只负责数据存储和统计
- 超过时间未记录 → 标记为 `missed`

### 6.4 AI 对话流式中转

```
手机 POST /api/chat/{sessionId}/send
  body: { "text": "我最近睡眠不好", "img_base64": null }
  响应类型: text/event-stream (SSE)
```

处理流程：

1. 中台存入 `chat_message`（role=user, msg_type=text）
2. 从 `ai_health_summary` 读取用户健康摘要
3. 组装 prompt（系统指令 + 健康摘要 + 近期对话历史 + 本次用户消息）
4. POST 云 AI `/chat/stream`，body: `{"text": "组装后的prompt", "img_url": "如有"}`
5. 云 AI 流式返回 `{"content": "..."}` → 中台通过 `SseEmitter` 逐段转发给手机
6. 收到 `[DONE]` → 拼接完整回复 → 存入 `chat_message`（role=assistant）→ 关闭 SSE

### 6.5 AI 健康摘要维护

触发时机：**健康档案发生变更时**（用户编辑个人信息、AI 对话中更新了健康信息、上传了新的医疗记录等）。

流程：
1. `health_profile` 更新 → 同时写入 `health_profile_history`
2. 中台异步调用云 AI，传入最新档案 + 近期关键数据 → AI 返回结构化摘要
3. 更新 `ai_health_summary.summary_json`

> 之所以选这个触发时机：档案变更代表用户健康状态有了新信息，摘要需要同步更新。而不是每次对话都更新，避免频繁调用 AI。

### 6.6 异常告警与 AI 主动对话

#### 场景

硬件检测到异常 → 手机端已在本地即时提醒用户 → 数据到达中台 → 中台判断是否需要唤起 AI 主动关怀

#### 中台的角色

手机端负责即时提醒（声音/震动/通知），中台负责**智能跟进**——让 AI 像一个关心你的健康助手一样主动找你聊天，提供专业分析和建议。

#### 防抖规则（避免反复唤起 AI）

| 规则 | 说明 |
|---|---|
| 同类型冷却 | 同一种 `alert_type` 触发 AI 后，30 分钟内不再重复触发 |
| 用户维度冷却 | 同一用户连续触发不同类型告警，至少间隔 5 分钟才触发下一次 AI |
| 严重度过滤 | 只有 `severity >= 2` 的告警才触发 AI 主动对话 |

> 冷却时间用 Redis 实现：`SET alert_cooldown:{userId}:{alertType} 1 EX 1800`，存在则跳过。

#### 触发流程

1. 收到 `flag != 0` 的 vital_sign → 生成 `health_alert`
2. 检查 Redis 冷却 key → 若冷却中则结束，记录告警但不唤起 AI
3. 通过冷却检查 → 设置 Redis 冷却 key
4. 创建 `chat_session`（source=`system_alert`, is_read=0）
5. 组装异常上下文（告警信息 + 近期生理数据 + 健康摘要）→ 调用云 AI
6. AI 回复存入 `chat_message` → 更新 `health_alert.triggered_session_id`
7. 手机端下次轮询 `/api/notify/poll` 时发现有新的 `system_alert` 会话 → 展示通知

#### 手机轮询机制

手机定期（如每 30 秒）调用 `GET /api/notify/poll`，中台返回：

```json
{
  "unread_alert_sessions": [
    {"session_id": 42, "title": "心率异常关怀", "preview": "检测到您的心率偏高..."}
  ],
  "unread_alerts_count": 3
}
```

> 用户点击后打开对话详情，可以继续和 AI 交流，此时和普通对话无异。

## 七、AI 对话协议

> 本节定义中台与云 AI 之间的交互约定。**中台只管组装请求和解析响应**，AI 内部怎么调工具/查知识库是 AI 队友的工作，中台不需要关心。

### 7.1 职责边界

```
中台的活：                           AI 队友的活：
├─ 组装 prompt（上下文拼接）          ├─ LangChain Agent 实现
├─ 调用 /chat/stream                 ├─ Tool 封装（查数据库、查知识库）
├─ SSE 透传给手机                    ├─ RAG 检索
├─ 解析 AI 回复中的结构化指令         ├─ Prompt 模板优化
└─ 执行指令（建计划、存医疗记录等）    └─ 模型选择与调优
```

> AI 如果需要查中台的数据（用户生理数据、用药记录等），通过 LangChain 的 Tool 机制调用中台暴露的 RESTful API。这些 Tool 怎么封装、怎么调用是 AI 队友负责的，中台只要确保 API 可用即可。

### 7.2 中台 → 云 AI：请求组装

中台调用 `POST /chat/stream`，body 中的 `text` 字段是组装好的完整 prompt。

#### 组装结构

```
[系统指令]
[健康摘要]
[对话历史（最近N轮）]
[本次用户消息]
```

#### 系统指令模板

```text
你是一位专业的个人健康管家AI助手。请基于用户的健康数据和对话内容，提供个性化的健康分析和建议。

规则：
1. 回答应基于用户的实际健康数据，不要凭空编造数据
2. 涉及严重健康问题时，建议用户及时就医，不要替代医生诊断
3. 语气亲切但专业
4. 如果你认为需要为用户创建或调整健康计划，请在回复末尾附上结构化指令（格式见下方）
5. 如果用户发送了医疗报告图片，请提取其中的关键信息并结构化整理

当你需要中台执行操作时，在回复的可见文本之后，另起一行以 ---ACTIONS--- 开头，用 JSON 格式描述操作：
---ACTIONS---
[{"action": "操作类型", "data": {操作数据}}]
```

#### 健康摘要注入

```text
## 当前用户健康概况
{ai_health_summary.summary_json 的内容}
```

#### 对话历史

取 `chat_message` 最近 10 条消息（5 轮对话），格式：

```text
## 对话历史
用户: xxxxx
助手: xxxxx
用户: xxxxx
助手: xxxxx
```

#### 最终请求示例

```json
{
  "text": "你是一位专业的个人健康管家AI助手...\n\n## 当前用户健康概况\n{摘要内容}\n\n## 对话历史\n用户: 昨天头疼了一下午\n助手: 头疼的位置是...\n\n## 用户消息\n我今天好多了，不过昨晚睡得不太好",
  "img_url": ""
}
```

### 7.3 云 AI → 中台：响应解析

#### 普通回复（无指令）

AI 流式返回纯文本，中台直接透传。最终完整回复示例：

```text
很高兴听到您头疼好转了。关于昨晚的睡眠问题，我注意到您最近几天的深睡时长都偏低...建议您今晚尝试睡前做10分钟深呼吸练习。
```

中台处理：直接存入 `chat_message`，无额外操作。

#### 带结构化指令的回复

AI 在可见文本之后附加操作指令：

```text
根据您的情况，我为您制定了一个改善睡眠的计划，您可以试着坚持一周看看效果：

1. 每天下午散步30分钟
2. 睡前1小时不接触电子屏幕
3. 晚上10点半之前上床

---ACTIONS---
[{"action": "create_plan", "data": {"title": "改善睡眠质量计划", "items": [{"desc": "每天下午散步30分钟", "category": "exercise"}, {"desc": "睡前1小时不接触电子屏幕", "category": "sleep"}, {"desc": "晚上10点半之前上床", "category": "sleep"}], "days": 7}}]
```

中台处理流程：

1. **流式阶段**：`---ACTIONS---` 之前的文本正常透传给手机
2. **流式阶段**：检测到 `---ACTIONS---` 标记后，后续内容不再透传（用户不需要看到 JSON）
3. **完成后**：解析 ACTIONS JSON，执行对应操作
4. **存储**：`chat_message.content` 只存可见文本部分（不含 ACTIONS）

#### 支持的 Action 类型

| action | 说明 | data 字段 |
|---|---|---|
| `create_plan` | 创建健康计划 | `title`, `items[]`（含 desc + category）, `days`（持续天数，可空） |
| `update_profile` | 更新健康档案 | 要更新的字段键值对，如 `{"weight_kg": 65.0, "lifestyle_info": "..."}` |
| `create_medical_record` | 从图片提取医疗记录 | `record_type`, `title`, `content`, `record_date` |

> **为什么用文本标记而不是单独接口**：AI 的操作建议是在对话上下文中产生的，和回复内容紧密关联。用文本内嵌的方式，AI 可以在一次回复中既给用户看文字，又告诉中台要执行什么操作，不需要额外的请求。

### 7.4 特殊场景的 prompt 组装

#### 睡眠分析（非对话，中台主动调用）

```json
{
  "text": "请分析以下用户的睡眠数据并给出评估：\n\n## 用户健康概况\n{摘要}\n\n## 昨晚睡眠数据\n入睡时间: 23:30\n起床时间: 07:15\n深睡(N3): 1.2小时 (15.6%)\nREM: 1.5小时 (19.5%)\n浅睡(N1+N2): 4.0小时 (51.9%)\n清醒: 1.0小时 (13.0%)\n\n整晚心率: 均值62, 最高85, 最低52\n整晚血氧: 均值97, 最高99, 最低94\n整晚体温: 均值36.5, 最高36.8, 最低36.2\n\n请给出：1.睡眠质量评分(1-10) 2.主要问题 3.改善建议",
  "img_url": ""
}
```

AI 回复直接存入 `sleep_session.ai_analysis`，不走 SSE 透传（因为不是对话场景，用户不在等）。

#### 异常告警触发的 AI 主动对话

```json
{
  "text": "你是一位专业的个人健康管家AI助手...\n\n## 当前用户健康概况\n{摘要}\n\n## 异常告警\n告警类型: 心率异常\n严重程度: 2\n描述: 用户心率持续偏高，最近一次记录心率102bpm，基线为72bpm\n告警时间: 2026-04-02 14:30\n\n## 近期生理数据\n最近30分钟心率: 98, 102, 99, 105, 101\n\n请以关心的语气主动询问用户的当前状况，并给出初步分析和建议。不要吓到用户，但要认真对待。",
  "img_url": ""
}
```

### 7.5 关于 AI 调用中台 API（Tool 调用）

AI 队友可能会把中台的某些查询接口封装为 LangChain Tool，让 AI 在对话中自主查询数据。这部分的分工：

| 中台做的 | AI 队友做的 |
|---|---|
| 确保 API 稳定可用 | 决定哪些 API 需要封装为 Tool |
| 提供 API 文档（Swagger） | 编写 Tool 的描述和调用逻辑 |
| 处理鉴权（API Key） | 在 LangChain Agent 中注册 Tool |

中台不需要为 Tool 调用做任何特殊处理——AI 调中台 API 和手机调中台 API 走的是同一套接口，只是鉴权方式不同（API Key vs JWT）。

## 八、已确定的设计决策（汇总）

| 决策项 | 结论 | 理由 |
|---|---|---|
| 图片存储 | **不存**。base64 临时传递，只存 AI 提取的文字 | 避免存储复杂度，base64 占空间，文字已足够 |
| 流式输出 | **全链路 SSE 透传**（云AI → 中台 SseEmitter → 手机） | 用户体验好，队友已验证 AI 侧流式可行 |
| 告警推送 | **手机轮询 + AI 主动对话**（中台不主动推送） | 手机端即时提醒已在本地完成，中台负责智能跟进 |
| AI 与中台交互方向 | **中台主动调用云AI**（非反向回调） | 简化架构，不需要 webhook |
| 健康摘要更新时机 | **健康档案变更时**触发 | 档案变更 = 新信息，适时更新且不频繁 |
| 健康计划打卡 | **复用 daily_log** 表，`log_type = 'plan_checkin'` | 避免加表，打卡逻辑轻量 |
